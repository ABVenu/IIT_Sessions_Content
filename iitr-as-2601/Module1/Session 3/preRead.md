## Introduction: What Is a Developer Workflow?

Imagine cooking without a routine.

Sometimes you chop first, sometimes you cook first, sometimes you forget ingredients.
The result is slow, messy, and inconsistent.

A **developer workflow** is the routine engineers follow to:

* Write code
* Test ideas
* Save progress
* Move forward safely

It’s not about speed—it’s about **control**.

## Why Workflow Matters

Without a workflow, small mistakes become big problems.

1. **You don’t lose work**
   Every meaningful change is saved and recoverable.

2. **You can experiment safely**
   Try ideas without fear of breaking everything.

3. **Others can understand your work**
   Your code history tells a clear story.

## From Known to New: Clicking → Commands

### The Comfortable Way

Most beginners rely on:

* Clicking folders
* Manually copying files
* Re-running code randomly

This works—until projects grow.

### The Professional Way

Developers use:

* **Commands** instead of clicks
* **Git** instead of manual backups
* Clear separation between experiments and production code

This workflow scales.

## Core Pieces of the Developer Workflow

### 1. Command Line Interface (CLI)

The **CLI** lets you talk to your computer using text commands.

* Faster than clicking for repetitive tasks
* Precise and predictable
* Used everywhere: servers, cloud, automation

*Think of it as giving direct instructions instead of gestures.*

### 2. Git: Tracking Your Work

**Git** is a version control system.

You’ll see three core actions:

* **init** → start tracking a project
* **add** → choose what changes to save
* **commit** → create a permanent snapshot

Each commit is a **checkpoint** you can return to.

### 3. Notebook vs Script Discipline

AI work often uses two formats:

#### Notebooks

* Interactive
* Great for exploration and learning
* Easy to experiment out of order

* Run top to bottom
* Predictable and repeatable
* Best for production and sharing

## How It All Fits Together

A common flow:

1. Explore ideas in a **notebook**
2. Convert stable logic into a **script**
3. Use **CLI** to manage files and run code
4. Save progress using **Git commits**

This keeps experiments flexible and results reliable.
